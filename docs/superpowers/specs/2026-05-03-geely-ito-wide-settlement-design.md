# 设计说明：吉利 ITO 结算单「宽表」双行表头（补充格式）

**日期：** 2026-05-03  
**状态：** 已评审（与业务方问答收敛），待实现  
**客户：** `geely_ito`（吉利 ITO）

## 1. 背景与目标

- 现有 `geely_ito` 使用**单行表头**结算列：`姓名`、`月度`、`月结算价`、`实际出勤天数`（规则见 `app/client_rules/rules/geely_ito.json`）。
- 业务新增一种**两行表头**、第二行为列名的宽表布局（分组合并行在上、列名在下），需在**不改变既有规则语义与默认路径**的前提下作为**补充**识别。
- 第一版仅扩展**结算区**解析；**差旅**仍只来自**独立差旅 Sheet**（与现规则一致）。

## 2. 识别策略（仅表结构，方案 A）

- **不**使用 Sheet 名或文件名作为第一道筛选（业务确认无法区分）。
- 在现有「按关键词查找结算 sheet + 表头扫描」流程中，对候选表头区域检测第二行（或算法最终认定的「列名行」）是否**同时存在**以下三列：
  - `姓名`
  - `月份`
  - `岗位结算单价`
- 若上述三列均存在，则进入**宽表分支**；并**必须**存在列 **`月度实际出勤天数`**，否则**报错退出**（禁止静默缺失出勤天）。
- 若三列未同时满足，则**完全**走现有单行表头逻辑（`月度` / `月结算价` / `实际出勤天数` 等），行为与线上一致。

### 2.1 歧义与优先级

- 旧模板使用 **`月度`**，宽表使用 **`月份`**；二者列名不同，正常文件不应同时满足两套「完整必填组」。
- 若极端情况下识别逻辑重叠：以「宽表签名」为准——即同时存在 `姓名` + `月份` + `岗位结算单价` + `月度实际出勤天数` 时走宽表；否则回落旧规则。

## 3. 列映射（宽表 → 内部标准列）

解析成功后，宽表须产出与现管道一致的 `settlement` 列：

| 内部列名 | 宽表表头（精确匹配列名） |
|----------|---------------------------|
| `name` | `姓名` |
| `ym` | `月份` |
| `monthly_price` | `岗位结算单价` |
| `actual_days` | `月度实际出勤天数` |

### 3.1 `月份` / `ym`

- 单元格格式：**`YYYY-MM`**（例如 `2021-10`）。
- 解析为与现逻辑一致的年月二元组（与现有 `geely_ito` 对 `月度` 的解析结果类型一致）。

### 3.2 含税口径

- `monthly_price`（来自 `岗位结算单价`）的含税处理与现 `ClientRule` 中 `geely_ito` 的 **`monthly_price_includes_vat`** 一致；**第一版不修改** `geely_ito.json` 中该字段的默认值或语义。

## 4. 非目标（第一版明确不做）

- **不**读入、**不**导出宽表中其余列（工号、部门、出差报销明细列、课外服务替换列、备注、合计公式列等）。
- **不**使用宽表内差旅相关列替代独立差旅 Sheet。
- **不**新增 `client_id`；仍为 `geely_ito`。

## 5. 差旅

- 与现 `geely_ito` 相同：独立「差旅」类 Sheet，列别名与必填组不变。

## 6. 错误处理

- 宽表签名（三列）满足但缺少 `月度实际出勤天数`：返回可读错误（建议 `code` 与现 `ConversionError` 体系一致），说明缺少列名。
- 表头扫描范围内无法稳定定位列名行：沿用现有结算表头查找与报错行为。

## 7. 测试建议

- 在 `scripts/smoke_test.py`（或等价烟测）中增加最小样例：
  - 两行表头 + 至少一行数据，`月份` 为 `YYYY-MM`；
  - 含标准差旅 Sheet；
  - 断言 `client_id == "geely_ito"`，且 `settlement` 含 `name, ym, monthly_price, actual_days` 且数值与样例行一致。

## 8. 后续可选（非本规格范围）

- 将宽表扩展列透传至导出、或校验第 30/31 列合计与系统重算一致性等，待单独需求与规格。
